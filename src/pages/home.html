From ask to backlog—by design

Change requests become stories—with every handoff explicit

Waterfall models the whole path in Git: change requests → requirements → use cases → tests and stories, with only the links your process allows. Everyone—PM, engineer, reviewer, agent—works off the same versioned graph, so “what we agreed” doesn’t drift from “what shipped.” The diagram below shows two parallel CRs converging through shared requirements, fanning into tests and backlog-ready stories, each tied to implementation.

Why it wins in real teams

Why teams pick Waterfall

  • Spec that ships with the product — change requests through stories live in Git next to your code: branch, review, and roll back intent the same way you do features.
  • Workflows you don’t have to re‑negotiate every sprint — the CLI encodes allowed edges (CR → RQ → UC → story, plus tests and implementation hooks) so the process stays consistent under pressure.
  • Cross‑cutting clarity, not wiki sprawl — horizontals (API, data, stack) attach across stories so shared models stay coherent instead of scattered across Confluence pages.
  • Agents on your rails, not a blank prompt — wire Cursor, Cline, or Claude Code with dry runs first; automation follows your spec tree instead of inventing requirements in chat.
