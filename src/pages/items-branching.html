Concept

Branching & change-request flow

How Git branches in a waterfall-spec repo line up with the change request (CR) lifecycle, which commands require develop vs a feature/CR-… branch, and how you can add artifacts by hand. Normative detail lives in CURSOR.md in your spec checkout; the CLI reference lists exact branchRequirement per command.

Two branch roles

  • develop — integration line for the spec tree. You do not author spec commits directly on develop (only merges in reviewed work). The CLI expects develop for opening a CR and for closing stories.
  • feature/CR-<NNN> or feature/CR-<NNN>-<slug> — one primary CR per branch; all lifecycle edits for that CR happen here until integration. Lifecycle prompts, sys start, horizontals, and cr finish run on this branch.
  • Other work (no single CR), e.g. typos in CURSOR.md or repo hygiene: use feature/<short-slug> from develop — not chore/* and not cr-… without the feature/ prefix (see CURSOR.md § Git workflow).

Flow for CR-scoped work

waterfall cr start (on develop) creates the CR artifact and checks out the matching feature/CR-… branch. You advance requirements and technical items on that branch, refresh horizontals as needed, then waterfall cr finish merges back into develop. 

flowchart LR
  dev([develop])
  feat[feature/CR-NNN-…]
  dev -->|cr start| feat
  feat -->|lifecycle · sys · horizontals| feat
  feat -->|cr finish| dev
  dev -->|story close| dev

Closing STORY-* folders is a develop operation in this model (see Story (STORY) and the CLI).

Commands by phase

All commands below assume your shell resolves the spec repo (e.g. --spec-root or .waterfall). Global flags such as --dry-run and --execute-agent apply as in the CLI reference.

Phase Branch waterfall commands
Open a CR develop cr start <description>
CR → requirements theme feature/CR-… cr to rq — or cr all (orchestrates the full CR-side pipeline in steps)
RQ → use cases feature/CR-… rq to uc — or rq all
UC → stories / tests feature/CR-… uc to story, uc to test, test to story — or uc all, test all
New subsystem (optional) feature/CR-… sys start <description>
Cross-cutting models feature/CR-… horizontal create story …, horizontal update <NNN>
Merge CR into develop feature/CR-… cr finish
Mark stories closed develop story close STORY-NNN, story close all

help, spec init, and help -o json do not enforce spec-repo branch checks the same way; see each command’s Branch line in the reference.

Manual addition (without the CLI)

You can create or edit markdown by hand as long as Git rules still hold:

  • Stay on the correct branch: develop only for merges and story-close policy; author CR-scoped spec changes on feature/CR-<NNN>-… created from up-to-date develop.
  • Allocate new typed ids by scanning existing folder names (next free index per kind), create folders per Repository structure, and keep ## Links and dependencies coherent — same as agent-driven edits.
  • You may run individual CLI steps later (e.g. hand-wrote CR and RQ, then waterfall rq to uc … on the feature branch) as long as the branch matches the command’s requirements.
  • For a new spec repo, waterfall spec init scaffolds the tree (and may initialize Git on develop); after that, follow CURSOR.md for branching.
