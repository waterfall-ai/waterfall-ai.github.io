Concept

Repository structure

A waterfall-spec checkout is a Git repo of markdown artifacts—not application source. Layout follows CURSOR.md in your tree; this page sketches the usual top-level shape after waterfall spec init.

Sketch of the tree

Example paths are illustrative; slugs and numeric suffixes follow CURSOR.md and existing folder names (next id = one past the highest index already present for that item type).

waterfall-spec/                    # your spec repo root
├── pom.json                       # project title (+ future metadata); from spec init
├── changerequests/
│   └── CR-001-example-feature/
│       └── CR-001.md
├── requirements/
│   └── RQ-101-goals/
│       ├── RQ-101.md
│       └── UC-201-main-flow/
│           └── UC-201.md
├── technical/
│   ├── SYS-301-core-api/
│   │   ├── SYS-301.md
│   │   └── _STORY-401-first-slice/
│   │       └── STORY-401.md
│   └── horizontals/
│       └── HOR-010-data-model/
│           └── HOR-010.md
├── tests/
│   └── TST-501-happy-path/
│       └── TST-501.md
├── prompts/                       # optional: workflow index + pointers (add manually)
├── doc/                           # optional: e.g. spec-structure.md (non-normative)
├── CURSOR.md                      # normative layout & rules for this repo
└── …

Top-level sections

Path / file Purpose Primary item types
pom.json Project metadata (required title from spec init; used e.g. for PDF export). —
changerequests/ Scoped product change: intent, impact, and traceability into requirements. CR
requirements/ Requirement themes; each theme folder holds its RQ doc and nested UC folders. RQ, UC
technical/ Technical subsystems; under each SYS live stories, bugs, and related technical docs. SYS, STORY, BUG
technical/horizontals/ Cross-cutting aggregates (data model, API model, stack) referenced by many stories. HOR
tests/ Acceptance test specifications (e.g. Gherkin-oriented) tied to UCs and stories. TST
prompts/ Optional spec-local workflow index and pointers into Waterfall CLI prompts (not created by spec init). —
doc/ Optional human-readable notes (non-normative); e.g. directory cheat sheets. —
CURSOR.md Normative “law” for layout, naming, branching, and graph rules in this repo. —

Authoritative detail lives in your spec’s CURSOR.md and in CLI reference / CLI module for commands that scaffold paths.
