Waterfall CLI

Run spec workflows from the terminal: branch-aware commands, dry-run plans, and real agent invocation when you choose.

Install

The CLI is published on npm as @pknx/waterfall-cli. The command you run is always waterfall (not the scoped package suffix). The unscoped package waterfall-cli is a different project.

Requirements: Node.js 20+ (22 recommended) and Git (used for spec init and lifecycle branches).

Global install (typical):

npm install -g @pknx/waterfall-cli
waterfall --help

Run without installing globally:

npx @pknx/waterfall-cli --help

First install may download heavier dependencies (e.g. diagram/PDF-related tooling). For every flag and command, see the CLI reference.

Getting started

After install, confirm the binary is on your PATH, then scaffold a new waterfall-spec repository with waterfall spec init. That command creates pom.json, the standard directories (changerequests/, requirements/, technical/, horizontals/, tests/, …), CURSOR.md, and an initial Git repo on branch develop. New item ids are derived from existing folders when you run scaffold commands, not from a separate counter file.

Rules spec init enforces

  • The target directory must be empty (no files yet).
  • There must be no pom.json or .git inside that directory yet.

Example: Bookstore-style lifecycle

Below is the core waterfall … sequence from the Waterfall project’s run-create-bookstore-spec.sh driver (minus a deliberate “second cr create must fail” check), stopping before PDF export and work-item sync. You only need Git, Node, and @pknx/waterfall-cli; work in a fresh directory that becomes the spec root after spec init. Configure backends and dry-run vs execute in .waterfall (see above). 

mkdir bookstore-spec && cd bookstore-spec
waterfall spec init --title "Bookstore" .

waterfall cr create \
  "Bookstore: product catalog, cart, checkout, and order history"

waterfall sys create \
  "Bookstore backend: catalog, cart, checkout, and order APIs."

waterfall sys create \
  "Bookstore storefront: browse, cart, checkout, account UI."

waterfall horizontal create story database-model \
  "Bookstore: collect and reconcile all database-related definitions from every STORY in this CR (entities, keys, persistence rules)."

waterfall horizontal create story api-model \
  "Bookstore: collect and reconcile public API shapes from every STORY (resources, methods, errors, versioning) for catalog, cart, and checkout."

waterfall horizontal create story technical-stack \
  "Bookstore: mongodb, docker, nextjs, REST API, OpenAPI spec, tailwindcss"

waterfall cr to rq \
  "Bookstore: RQ themes for catalog search, cart/checkout, and account orders."

waterfall rq to uc \
  "Bookstore: per RQ theme, add UC markdown with TC sections (browse, cart, checkout)."

waterfall uc to story \
  "Map bookstore UCs to STORY folders under technical/SYS areas."

waterfall uc to test \
  "Bookstore: TST specs for catalog, cart, and checkout flows."

waterfall test to story \
  "Align STORY work with TST-linked acceptance for Bookstore flows."

waterfall horizontal update 001
waterfall horizontal update 002
waterfall horizontal update 003

waterfall cr finish

waterfall cr create \
  "Bookstore: modern responsive Amazon-like ecommerce storefront (product grid, search, cart/checkout UX)."

waterfall cr to rq \
  "CR2 UI only: RQ themes for responsive layout, product grid/detail, search and filters, cart/checkout polish, accessibility — under existing Bookstore SYS."

waterfall rq all \
  "Bookstore: execute full RQ pipeline for CR2 — UC per theme, stories, TST linkage, test-to-story alignment."

waterfall horizontal update 001
waterfall horizontal update 002
waterfall horizontal update 003

waterfall cr finish

The integration script in the repo runs these via node …/bin/waterfall.mjs from a parent folder with the spec in ./spec; here everything uses . as the spec root after cd bookstore-spec. Branch rules still apply (e.g. cr finish merges the feature branch to develop before the second cr create).

What to do next

  • Run waterfall help from inside or above the spec tree (the CLI resolves the spec root automatically).
  • Add a .waterfall file in the spec repo or your home directory for defaults (backend, dry-run, log level); see configuration in the reference.
  • For a reference .waterfall used with this scenario, see tests/bookstore/.waterfall on GitHub.
  • On develop, open work with waterfall cr create "…" (see general examples).

What it does

  • Resolve your spec root via .waterfall, flags, waterfall.json, or walk-up discovery.
  • CR lifecycle: cr create, stepwise edges (cr to rq, …), cr all, cr finish.
  • Subsystem, bug, and horizontal scaffolds: sys create, bug create, bug close, horizontal create / horizontal update; story close for delivered stories.
  • Backends: cline-ollama, Cursor, Claude Code—with embed or reference prompt delivery.

Configuration (.waterfall)

The usual place to tune the CLI is a .waterfall file in your spec repository root (and optionally ~/.waterfall for personal defaults). On first use, the tool can create a commented example; values there cover things like default backend, whether to dry-run agent steps, log verbosity, spec root override, prompt delivery, and work-item sync blocks. Precedence is flags → environment → ./.waterfall → ~/.waterfall → defaults—see CLI reference · Configuration files for the full format and field list.

Non-reserved UPPER_SNAKE keys in .waterfall are passed through to agent subprocesses (e.g. API keys). For CI or one-off overrides you can still use WATERFALL_* env vars or flags; the complete env catalogue lives under Environment variables in the reference.

Enhancing PDF export quality

waterfall export pdf (see the CLI reference) can use two pipelines. By default it uses bundled pandoc-wasm plus headless Chromium via Puppeteer—no extra installs, good for a quick artifact. For sharper typography, embedded fonts, and typical Pandoc/LaTeX output, switch to system Pandoc with a PDF engine (defaults to XeLaTeX for Unicode in spec text).

In your spec repo’s .waterfall set use_native_pandoc=true. Optional tuning: WATERFALL_PANDOC (path to the pandoc binary), WATERFALL_PANDOC_PDF_ENGINE, and WATERFALL_PANDOC_ARGS—documented under environment variables and configuration.

Example: macOS with Homebrew

Install Pandoc:

brew install pandoc

XeLaTeX ships with a TeX distribution. A common lightweight route is BasicTeX (smaller download than full MacTeX); after install, open a new terminal and ensure xelatex is on your PATH (BasicTeX may need extra packages via tlmgr if Pandoc complains about missing LaTeX pieces). 

brew install --cask basictex
# New shell, then e.g.:
sudo tlmgr update --self
sudo tlmgr install collection-fontsrecommended

Alternatively install MacTeX if you prefer an all-in-one distribution (brew install --cask mactex — large install). Then enable native Pandoc in .waterfall and run waterfall export pdf from a clean Git working tree at the spec root.
